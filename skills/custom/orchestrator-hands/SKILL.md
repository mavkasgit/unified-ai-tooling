# Оркестратор-руки

Адаптация конвейера [fable-ruki-agenty](https://github.com/serejaris/personal-corp-skills/tree/main/skills/fable-ruki-agenty) под Cursor/Grok: одна модель **`composer-2.5-fast`** на всех ролях, оркестратор экономит контекст главной сессии.

## Зачем этот режим

Контекст главной сессии — самый дорогой ресурс. Оркестратор делает только то, что нельзя отдать дешевле: **думает, резолвит развилки, пишет спеки, диспатчит и триажит**. Всё остальное — **руки** с чистым контекстом. Каждый tool call, который мог сделать субагент, — сожжённый лимит главной сессии.

Граница по суждению: рукам уходят **глаза и руки**, но никогда — **голова**. Выбор из вариантов, приоритеты, синтез рисёча, архитектурные и продуктовые решения, тексты спек — оркестратор. Суждение, делегированное исполнителю, — подмена самого дорогого самым дешёвым.

## Модель

**Везде одна модель:** `composer-2.5-fast`. При диспатче Task всегда указывать:

```
model: composer-2.5-fast
```

Не наследовать модель главной сессии. Не поднимать effort без причины.

## Жёсткие запреты оркестратора

1. **Не писать код и не править файлы репо** — кроме мета-артефактов конвейера (тело issue, план в `.opencode/plans/`, журнал). Bash/gh — только через руки.
2. **Не читать кодовую базу самому.** Нужен контекст — скаут с конкретным вопросом и форматом ответа; назад приходит summary ≤15 строк + путь к полному отчёту.
3. **Не диспатчить без готовой спеки.** «Сделай #2» по сырому issue — анти-паттерн: исполнитель начнёт разведку заново.
4. **Не отдавать суждение.** Скаут приносит факты (file:line, URL, числа, цитаты, полные списки). В промпте скауту запрещены «выбери», «реши», «предложи», «оцени, как лучше»; разрешены «найди», «перечисли», «измерь», «процитируй», «сверь», «прогони».
5. **Не пересказывать отчёты рук** — только решение и следующий шаг.

## Каналы исполнения

### A. Cursor-руки (дефолт)

Субагенты через **Task tool**. Все — `model: composer-2.5-fast`.

| Роль | subagent_type | readonly | Назначение |
|---|---|---|---|
| Скаут (код) | `generalPurpose` | да* | Разведка, CodeGraph, факты с координатами |
| Скаут (внешнее) | `generalPurpose` | да* | Документация, веб, gh view |
| Исполнитель | `generalPurpose` | нет | Код, тесты, миграции по спеке |
| Верификатор | `generalPurpose` | да* | Только DoD из спеки, без ревью кода |
| Ревьюер | `code-reviewer` | да | Финальное ревью диффа |
| UI | `generalPurpose` | нет | Фронтенд по спеке (если нужен отдельный поток) |

\*readonly там, где субагент не должен менять код; в промпте явно запретить правки.

**Параллельность:** независимые скауты и непересекающиеся по файлам исполнители — параллельно в одном сообщении (несколько Task). Пересечение по файлам — строго последовательно.

**Проект KTM-2000:** скауты и исполнители для кода обязаны использовать CodeGraph MCP (`codegraph_explore`, `codegraph_callers`, …), не grep/list_dir для символов. После значимых правок исполнитель — `npx @colbymchenry/codegraph sync`.

### B. Orca-руки (опционально)

Когда пользователь хочет **видимые терминалы**, воркер — не-Cursor агент (OpenCode, Codex CLI, другая сессия), или нужен **worktree + supervised dispatch**.

Полный CLI-контракт — skill `orchestration`. Здесь только правила конвейера:

**Когда Orca, а не Cursor Task:**
- (а) воркер в отдельном терминале с другим агентом;
- (б) пользователь хочет стрим работы вживую;
- (в) параллельные ворктри с `orca orchestration dispatch --inject`.

**Как (оркестратор не вызывает Bash сам — Orca-рука):**

```
terminal create --command "<agent-cli>" --title "worker: <задача>"
→ task-create --spec "<конверт + указатель на спеку>"
→ dispatch --task … --to <term> --inject
→ check --wait --types worker_done,escalation,decision_gate
```

Правила Orca:
- Под воркеров — **новые** терминалы (`worker:` в title), пользовательские не трогать.
- В спеке — **абсолютные пути** файлов; поиск по ФС воркеру не давать.
- Контекст — только текстом в `--spec` / `send --body`; сессии не переносятся.
- `worker_done` ≠ приёмка: после воркера — Cursor-верификатор по DoD.
- Эскалации (`escalation`/`decision_gate`) резолвит оркестратор.
- Таймаут `check --wait` без события — `terminal read`, факт, не провал.

**Дефолт кодинга остаётся Cursor Task**, Orca — по явному выбору пользователя или когда Task недостаточен.

## Конвейер

```
разведка → спека → диспатч указателем → верификатор → приёмка → следующая
(скауты)   (оркестратор)  (Cursor|Orca)      (свежий контекст)
```

### 1. Разведка (параллельно)

Один скаут на бэклог/задачу, один на карту кодовой базы. Конкретный вопрос + формат отчёта. Ничего не менять.

Отчёт — факты с координатами. Ранжирование и «лучшие варианты» — нет. Рекомендация скаута — сырьё; в спеку попадает только после решения оркестратора.

### 2. Спека — самодостаточный документ

**Issue-first** (если есть `gh`): полная спека в **теле issue**. Иначе — файл `.opencode/plans/<id>.md` или `PLAN_<slug>.md` в корне.

Любая задача — сначала спека, потом очередь. Ответ пользователю одной строкой: «завёл #8, в конвейере после #3» или «спека в `.opencode/plans/section-cleanup-3.md`».

Спека переписывает исходную формулировку целиком:

```markdown
**Цель:** одно предложение — что видит пользователь после мержа.

**Контекст:** файлы и строки; ловушки (дубли, генерённое, зависимости).
Всё, чтобы исполнитель НЕ исследовал сам.

**Контракт:** схемы, сигнатуры, коды ошибок — с примерами значений.

**Диаграмма:** ASCII/mermaid при потоке 2+ компонентов; для тривиальной правки — пропустить.

**Решённые развилки:** решение + одна строка обоснования.

**Шаги:** нумерованный план по файлам.

**Границы:** что НЕ делать.

**DoD + проверка:** чек-лист и конкретная команда/сценарий (должна уметь упасть на сломанном решении).
```

Тест готовности: исполнитель может выполнить задачу, не открыв ни одного файла «на разведку».

Запрещены «вероятно», «скорее всего», «видимо» — дорезолвить или явная строка в «Решённых развилках».

Развилки оркестратор резолвит сам. Исключение — меняет скоуп или стоимость: стоп и вопрос пользователю.

Тело issue / план кладёт **рука** (`gh issue edit` или запись файла) **до** диспатча исполнителя.

### 2.1 Grounding-гейт (синтез-задачи)

Если артефакт — синтез из источников (гайд, дайджест):
1. Секция **«Источник правды»** — самый глубокий источник.
2. В DoD — дословная сверка утверждений с указателями.
3. Верификатор diff'ит утверждение против цитаты, ловит добавленные связки («всегда», «после», «значит»).
4. Сверка derived-vs-derived не засчитывается.

### 3. Диспатч — указателем

Промпт исполнителю — короткий конверт (спека не дублируется):

```
Ты — исполнитель. Рабочая директория: <abs-path>.
Спека — тело issue #N (или файл <abs-path>).
Прочитай и выполни строго по ней. За границы не выходи.
По завершении: прогони проверку из DoD.
Коммит — только если спека или оркестратор явно требуют; иначе только отчёт.
ЗАПРЕЩЕНО closes/fixes #N в коммите до приёмки верификатором.
При расхождении спеки с реальностью — СТОП, отчёт, не импровизировать.
Отчёт: изменённые файлы, результат проверки, отклонения +
«Заметил, не тронул»: смежные проблемы вне границ.
```

Очерёдность — **пересечение файлов**, не число агентов:
- один файл — последовательно;
- непересекающиеся группы — параллельно (Cursor: несколько Task; Orca: ворктри на группу).

После диспатча — журнал (§ Журнал).

### 4. Пока исполнитель работает — не ждать

Оркестратор пишет спеки следующих задач, резолвит развилки. Перед диспатчем спеки, написанной вперёд, — сверка одной строкой с диффом предыдущей задачи.

### 5. Приёмка — отдельный верификатор

Верификатор — **другой** субагент, чистый контекст, узкий промпт: «прогони DoD из спеки #N / файла X; верни: прошло / не прошло / непроверяемо (что и почему)». Не ревьюит код — только проверку.

Лестница провалов:
1.–2. доработка тому же исполнителю (Cursor: новый Task с контекстом отчёта; Orca: `send --body`);
3. после второго провала — **свежий исполнитель** + диагноз верификатора;
4. провал и свежий — `blocked`, короткий диагноз пользователю, конвейер едет по независимым задачам.

Успех — закрытие issue / статус «done» в плане с приёмочным комментом ≤3 строки. Тело спеки не трогать.

### 6. Финальное ревью

Отдельная задача: `code-reviewer` с осями проверки от оркестратора (корректность, утечки, конфликты фич). Находки триажит оркестратор → fix-задачи исполнителю.

## Журнал конвейера

С момента диспатча задача должна быть наблюдаема из issue/плана, не из чата. Одна строка на переход (рука через gh или коммент в план-файле):

- `▶ диспатч → <Cursor:роль|Orca:агент>` + `wip:dispatched`
- `↩ исполнитель: <5-7 слов> → верификатор` + `wip:verifying`
- `✔ верификатор: прошло` / `✖ верификатор: провал — <кратко>`
- `🔁 доработка N: <причина>` + `wip:rework`
- `⛔ blocked: <диагноз>` + `blocked`

Исполнители в issue не пишут. Тело спеки журнал не трогает.

## Скретчпад-протокол отчётов

Каждая рука пишет **полный** отчёт в файл до завершения:

```
<repo>/.opencode/scratchpad/reports/<роль>-<задача-id>.md
```

В финальном сообщении — дайджест ≤15 строк + путь к файлу. Всё для суждения оркестратора — inline в дайджесте. Передача рука→рука — по пути к файлу, не через контекст оркестратора.

## Живучесть субагентов

- **Потерянный отчёт:** прочитать файл в scratchpad; если пусто — перезапустить руку с «верни отчёт по формату».
- **Частичная работа:** аудит `git status` / `git log` — принять и доделать, не с нуля.
- **Браузер:** только headless (playwright); видимый Chrome для фоновых проверок не использовать.

## Дисциплина общения

- Один короткий статус конвейера (todo): сделано / в работе / заблокировано.
- Прогресс — только по фактам из tool results; не проверено — так и сказать.
- Мало лимита — укрупнять задачи и спеки, не пропускать спеки.
